# 🧪 Complete API Testing Guide - Profile Management, Family Management, Delegates, Workers & Clubhouse Flow

## Overview

This guide provides comprehensive testing instructions for Profile Management, Family Management, Delegates, Workers, and Clubhouse Access Control Flow. All tests assume:

- **Base URL:** `http://localhost:3000`
- **Authentication:** `Authorization: Bearer <jwt-token>` header
- **Admin Token:** Required for admin-only endpoints
- **Owner Token:** Required for owner-specific endpoints
- **Tenant Token:** Required for tenant-specific endpoints
- **Family Token:** Required for family member-specific endpoints

## 📋 Profile Management - Core Principles

### **Profile = Identity (Singular, Self-Owned)**
A profile represents WHO YOU ARE and is completely independent of units, leases, or family relationships.

**✅ Profile Includes:**
- Identity & personal data (Full name, National ID, Date of birth, Gender, Nationality)
- Contact information (Phone, Email, Emergency contact)
- Authentication-adjacent (User ID, Role(s), Account status)
- Files (profilePhotoId, nationalIdFileId - 1-to-1 max)

**❌ Profile Does NOT Include:**
- Family members ❌
- Marriage certificates ❌
- Lease contracts ❌
- Unit access ❌

**🔑 Key Rule:** A profile can exist without a unit, without family, without leases. If your system breaks when a user has no unit → bad architecture.

**🎯 Profile Management is Singular & Self-Owned:**
- Users can only manage their OWN profile
- No one else can edit your profile (not even admins for personal data)
- Admins can only manage account status and roles
- Profile is boring by design - it's just your identity

### **4.2 Enhanced Lease Termination Cascade**
**Endpoint:** `POST /leases/{leaseId}/terminate`

**Headers:**
```
Authorization: Bearer <admin-or-owner-token>
Content-Type: application/json
```

**Request Body:**
```json
{
  "reason": "End of contract period",
  "terminationDate": "2024-06-01T00:00:00.000Z"
}
```

**✅ Success Response (200):**
```json
{
  "id": "gg0e8400-e29b-41d4-a716-446655440011",
  "status": "TERMINATED",
  "endDate": "2024-06-01T00:00:00.000Z"
}
```

**✅ Cascade Effects (Status-Only Updates):**
1. **Lease Status**: ACTIVE → TERMINATED
2. **Unit Access**: TENANT access → EXPIRED
3. **ResidentUnit**: Marked inactive (NOT deleted)
4. **Family Deactivation**: If tenant has NO other active units → Family status → INACTIVE
5. **Tenant User Status**: If tenant has NO active leases AND NO owned units → User status → INACTIVE
6. **Email Notification**: Sent to tenant with termination details

4. Add another family member → Family gets access to BOTH units
5. Terminate lease on Unit A → Family stays ACTIVE (tenant has Unit B)
6. Terminate lease on Unit B → Family becomes INACTIVE (tenant has no units)

**Expected Behavior:**
- Family members stay active as long as tenant has ANY active units
- Only when tenant loses ALL units do family members become inactive
- No hard deletions - only status transitions

### **5.3 Tenant Deactivation Rules**

**Tenant User Status Transitions:**
1. **Active Lease + Owned Unit**: User status = ACTIVE
2. **Active Lease Only**: User status = ACTIVE
3. **Owned Unit Only**: User status = ACTIVE
4. **No Active Leases + No Owned Units**: User status = INACTIVE

**Example Flow:**
```json
// Initial State
{
  "userStatus": "ACTIVE",
  "leaseCount": 1,
  "ownedUnits": 0
}

// After Lease Termination (Last Unit)
{
  "userStatus": "INACTIVE",
  "leaseCount": 0,
  "ownedUnits": 0
}
```
