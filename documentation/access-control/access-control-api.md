# Access Control (QR Codes) API

This document covers the current behavior of the Access Control module (`src/modules/access-control`).

## Business Scope

Purpose:
- Residents generate QR codes for self / visitors / deliveries / workers / service providers / rideshare.
- QR codes are issued, stored, tracked, and auditable in our DB.
- QR validation/enforcement is handled by HikCentral (we do not validate scans).

Out of scope (current backend):
- Validating QR scans at gates
- Enforcing “one-time use” beyond storing status/metadata
- Receiving “scan used” events from HikCentral (webhooks/polling not implemented)

## Auth

All routes require a valid JWT (`Authorization: Bearer <token>`).

Additional service-level authorization rules:
- Generating a QR: requester must have ACTIVE `UnitAccess` on the unit **and** `UnitAccess.canGenerateQR = true`.
- Listing QRs:
  - If `unitId` is provided:
    - Unit owner (`UnitAccess.role=OWNER`) can see all QRs for that unit.
    - Non-owner can only see QRs they generated for that unit.
  - If `unitId` is not provided: lists only QRs generated by the requester.
- Revoking a QR:
  - The generator can revoke their own QR.
  - Unit owner can revoke any QR for that unit.

## Data Model

Existing Prisma models/enums (already in `prisma/schema.prisma`):
- `UnitAccess.canGenerateQR` controls who can generate QRs for a unit.
- `AccessQRCode` stores generated QR metadata + HikCentral `qrId` token.
- `AccessGrant` stores temporary access permissions for types that require it.
- `QRType` enum: `SELF`, `VISITOR`, `DELIVERY`, `WORKER`, `SERVICE_PROVIDER`, `RIDESHARE`
- `AccessGrantPermission` enum: `ENTER`, `WORK`, `DELIVER`
- `AccessStatus` enum is reused for QR lifecycle: `ACTIVE`, `EXPIRED`, `USED`, `REVOKED`, `CANCELLED`

Notes about current implementation:
- `AccessQRCode.status` is set to `ACTIVE` on creation.
- `EXPIRED` is applied lazily during listing (see `GET /access-qrcodes` behavior).
- `USED` is not set automatically (no HikCentral event ingestion yet).
- For grant-required QR types, a new `AccessProfile` is created as a placeholder identity to satisfy the `AccessGrant.accessProfileId` relation.

## Endpoints

1. `POST /access-qrcodes`
2. `GET /access-qrcodes`
3. `PATCH /access-qrcodes/:id/revoke`

## 1) POST /access-qrcodes

Generates an access QR code, stores metadata in DB, and returns the QR payload for the frontend.

### Content Types

- `application/json`

### Request Body

`CreateAccessQrCodeDto` fields:
- `unitId` (uuid, required)
- `type` (QRType enum, required)
- `visitorName` (string, required for `VISITOR`)
- `validFrom` (ISO string, optional)
- `validTo` (ISO string, optional)
- `permissions` (AccessGrantPermission[], optional)
  - If omitted, defaults per type (see rules below).
- `gates` (string[], optional)
- `notes` (string, optional)

Example:
```json
{
  "unitId": "107565e6-a9ad-4d0e-b4f3-322b9c20c37f",
  "type": "VISITOR",
  "visitorName": "Ahmed Visitor",
  "validFrom": "2026-02-03T10:00:00.000Z",
  "validTo": "2026-02-03T11:00:00.000Z",
  "permissions": ["ENTER"],
  "gates": ["GATE_A"],
  "notes": "Guest coming for a visit"
}
```

### Default Validity (if `validFrom`/`validTo` omitted)

- `SELF`: 5 minutes
- `DELIVERY`: 30 minutes
- Others: 1 hour

If only `validFrom` is provided, `validTo` is computed from defaults.  
If `validTo` is provided, it must be strictly after `validFrom`.

### Default Permissions (if `permissions` omitted)

- `DELIVERY` → `["DELIVER"]`
- `WORKER` / `SERVICE_PROVIDER` → `["WORK"]`
- `SELF` / `VISITOR` / `RIDESHARE` → `["ENTER"]`

### Validations / Business Rules

Returns 400/403/404/409/502 with these rules:
- 400 if `type` is missing/invalid
- 400 if `VISITOR` and `visitorName` is missing
- 400 if `validTo <= validFrom`
- 403 if requester has no ACTIVE `UnitAccess` on `unitId`
- 403 if requester has ACTIVE `UnitAccess` but `canGenerateQR=false`
- 409 if `QR_ENFORCE_SINGLE_ACTIVE` is enabled and an ACTIVE (not expired) QR already exists for the same `(unitId, type)`
- 502 if HikCentral call fails (non-mock mode)

### Side Effects

- Optionally creates an `AccessGrant`:
  - Created for: `VISITOR`, `DELIVERY`, `WORKER`, `SERVICE_PROVIDER`
  - Not created for: `SELF`, `RIDESHARE`
- Calls HikCentral “create QR” integration
- Creates `AccessQRCode` with:
  - `qrId` from HikCentral
  - `status=ACTIVE`
  - `generatedById`, `unitId`, `type`, `validFrom`, `validTo`, `visitorName`, `gates`, `notes`
- If HikCentral fails after grant creation, the service best-effort deletes the created `AccessGrant` and the placeholder `AccessProfile`.

### Response

```json
{
  "qrCode": {
    "id": "<db id>",
    "qrId": "<hikcentral id/token>",
    "type": "VISITOR",
    "generatedById": "<user id>",
    "unitId": "<unit id>",
    "accessGrantId": "<grant id or null>",
    "visitorName": "Ahmed Visitor",
    "validFrom": "2026-02-03T10:00:00.000Z",
    "validTo": "2026-02-03T11:00:00.000Z",
    "gates": ["GATE_A"],
    "scans": 0,
    "status": "ACTIVE",
    "notes": "Guest coming for a visit",
    "createdAt": "2026-02-03T10:00:01.000Z",
    "updatedAt": "2026-02-03T10:00:01.000Z"
  },
  "qrImageBase64": "<base64 string>"
}
```

### Test Cases

Happy path:
- Generate `SELF` with defaults (no dates)
- Generate `VISITOR` with visitorName + explicit validity
- Generate `DELIVERY` with default permissions

Failure:
- `VISITOR` without `visitorName` → 400
- `validTo <= validFrom` → 400
- No `UnitAccess` on unit → 403
- `canGenerateQR=false` → 403
- Duplicate QR for same type/unit when enforced → 409

## 2) GET /access-qrcodes

Lists QR codes.

### Query Params

`ListAccessQrCodesDto` fields:
- `unitId` (uuid, optional)
- `includeInactive` (boolean, optional; default `false`)

Examples:
- `GET /access-qrcodes` (list only my ACTIVE QRs)
- `GET /access-qrcodes?includeInactive=true` (list my ACTIVE + EXPIRED/REVOKED/etc.)
- `GET /access-qrcodes?unitId=<unitId>` (list ACTIVE QRs scoped to unit rules)
- `GET /access-qrcodes?unitId=<unitId>&includeInactive=true`

### Behavior

- Before returning results, the service performs a best-effort status update:
  - Any `AccessQRCode` in-scope with `status=ACTIVE` and `validTo <= now` is updated to `status=EXPIRED`.
- Scoping:
  - With `unitId`:
    - OWNER sees QRs for the unit.
    - Non-owner sees only their own QRs for that unit.
  - Without `unitId`:
    - Lists only QRs created by the requester.

### Response

Returns an array of `AccessQRCode` rows (sorted by `createdAt desc`).

### Test Cases

- List my active QRs
- List my active + inactive QRs
- List by unit as owner vs non-owner
- Verify that expired ACTIVE rows get converted to `EXPIRED` on read

---

## 3) PATCH /access-qrcodes/:id/revoke

Revokes an ACTIVE QR code (DB-only in current implementation).

### Authorization

- QR generator can revoke their own QR.
- Unit owner can revoke any QR for their unit.

### Behavior

- 404 if QR not found
- 400 if QR is not `ACTIVE`
- 403 if requester is neither the generator nor the unit owner
- Updates `AccessQRCode.status` to `REVOKED`

### Response

Returns the updated `AccessQRCode` row.

### Test Cases

- Generator revokes their QR
- Owner revokes another user’s QR for their unit
- Non-owner/non-generator revokes → 403
- Revoking an already REVOKED/EXPIRED QR → 400

## HikCentral Integration (Current vs Required)

### What’s Implemented Today

`src/modules/access-control/hikcentral/hikcentral-qr.service.ts` supports:
- Mock mode (no real HikCentral call) when:
  - `HIKCENTRAL_MOCK=true`, OR
  - `HIKCENTRAL_BASE_URL` is not set
- “Real” mode using a generic `fetch` POST:
  - URL: `{HIKCENTRAL_BASE_URL}{HIKCENTRAL_QR_CREATE_PATH || "/qr-codes"}`
  - Auth header: `Authorization: Bearer {HIKCENTRAL_API_KEY}` (if provided)
  - Expects response fields: `qrId` and `qrImageBase64` (or nested under `data`)

This is a placeholder contract until you align it with HikCentral’s real API.

### What You Still Need to Wire HikCentral Completely

1) Official HikCentral API contract for “Create QR”
   - Endpoint, required payload, response schema, timezone rules
2) Official authentication method
   - API key vs session token vs OAuth2 vs HMAC signing, required headers
3) Mapping rules
   - How `unitId` maps to HikCentral entities
   - How `permissions` maps to HikCentral access levels
   - How `gates[]` maps to HikCentral door/gate identifiers
4) Implement “Revoke/Disable QR” in HikCentral (if supported)
   - Add a `revokeQrCode(qrId)` call and wire it into `PATCH /access-qrcodes/:id/revoke`
5) Optional: usage reporting
   - Webhook endpoint or polling to mark QRs as `USED`
6) Optional: persist HikCentral raw payload
   - If you need “store HikCentral payload”, add a JSON column in `AccessQRCode` (migration) and store `raw`

## Local / Postman Testing

You can test this module immediately without HikCentral wiring:
- Set `HIKCENTRAL_MOCK=true` (or leave `HIKCENTRAL_BASE_URL` unset).
- Ensure your test user has ACTIVE `UnitAccess` for the unit with `canGenerateQR=true`.
- Use Postman against:
  - `POST /access-qrcodes`
  - `GET /access-qrcodes`
  - `PATCH /access-qrcodes/:id/revoke`

## Configuration

Environment variables:
- `HIKCENTRAL_MOCK` (default: false, but mock is auto-enabled if no base URL)
- `HIKCENTRAL_BASE_URL` (required for real HikCentral calls)
- `HIKCENTRAL_QR_CREATE_PATH` (default: `/qr-codes`)
- `HIKCENTRAL_API_KEY` (optional placeholder auth mechanism)
- `QR_ENFORCE_SINGLE_ACTIVE` (default: true; set to `false` to allow multiple ACTIVE QRs per type/unit)
