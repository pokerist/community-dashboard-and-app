# Access Control Module (QR Codes) (`src/modules/access-control`)

## What this module is responsible for

This module issues and manages access QR codes for a unit:

- Generate a HikCentral-backed QR code and persist it in the DB.
- List QR codes (scoped by unit and/or creator).
- Revoke an active QR code (DB status update).

QR scan validation/enforcement happens in HikCentral; this backend currently does not receive scan events.

## Authentication / authorization

All routes require a valid JWT (`JwtAuthGuard`).

Service-level authorization rules:

- **Generate QR**: requester must have ACTIVE `UnitAccess` on the unit AND `UnitAccess.canGenerateQR = true`.
- **List QRs**:
  - If `unitId` is provided:
    - Unit owner (`UnitAccess.role = OWNER`) can see all QRs for the unit.
    - Non-owner can only see QRs they generated for that unit.
  - If `unitId` is not provided: lists only QRs generated by the requester.
- **Revoke QR**:
  - The generator can revoke their own QR.
  - The unit owner can revoke any QR for their unit.

## Key data models (Prisma)

From `prisma/schema.prisma`:

- `AccessQRCode`
  - Stores generated QR metadata + HikCentral `qrId` token.
  - Uses `AccessStatus` for lifecycle: `ACTIVE`, `EXPIRED`, `REVOKED`, etc.
- `AccessGrant`
  - Created for QR types that require explicit permissions (visitor/delivery/worker/service provider).
- `AccessProfile`
  - For grant-required QR types, an `AccessProfile` row is created to satisfy the `AccessGrant.accessProfileId` relation.
  - Current implementation uses a generated `nationalId` like `QR-<uuid>`.
- `UnitAccess.canGenerateQR`
  - Controls who can generate QRs for a unit.

Enums:

- `QRType`: `SELF`, `VISITOR`, `DELIVERY`, `WORKER`, `SERVICE_PROVIDER`, `RIDESHARE`
- `AccessGrantPermission`: `ENTER`, `WORK`, `DELIVER`
- `AccessStatus`: reused for QR lifecycle

Note: for contractor/worker workflows (persistent worker identities + delegate authorization), prefer the Workers module (`documentation/workers/README.md`) and `POST /workers/:id/qr` over `POST /access-qrcodes` with `type=WORKER`.

## QR lifecycle behavior

- On creation, QRs are stored with `status = ACTIVE`.
- On listing (`GET /access-qrcodes`), the service lazily expires QRs:
  - any in-scope `AccessQRCode` with `status=ACTIVE` and `validTo <= now` is updated to `status=EXPIRED`.
- `USED` is not set automatically (no webhook/polling ingestion from HikCentral yet).
- Revoke updates the DB row to `status=REVOKED` (HikCentral revoke is not implemented).

## API surface (controller)

Base route: `/access-qrcodes`

- `POST /access-qrcodes` (generate)
- `GET /access-qrcodes` (list)
- `PATCH /access-qrcodes/:id/revoke` (revoke)

## Create QR behavior (defaults and validations)

Defaults:

- If `validFrom`/`validTo` are omitted:
  - `SELF`: 5 minutes
  - `DELIVERY`: 30 minutes
  - Others: 1 hour
- If `permissions` is omitted:
  - `DELIVERY` -> `["DELIVER"]`
  - `WORKER` / `SERVICE_PROVIDER` -> `["WORK"]`
  - `SELF` / `VISITOR` / `RIDESHARE` -> `["ENTER"]`

Validations:

- `VISITOR` requires `visitorName`.
- `validTo` must be strictly after `validFrom`.
- If `QR_ENFORCE_SINGLE_ACTIVE` is enabled (default), creating a QR conflicts (409) when an ACTIVE, not-yet-expired QR already exists for the same `(unitId, type)`.

## HikCentral integration

Implementation: `src/modules/access-control/hikcentral/hikcentral-qr.service.ts`.

Modes:

- Mock mode is enabled when:
  - `HIKCENTRAL_MOCK=true`, OR
  - `HIKCENTRAL_BASE_URL` is not set
- Real mode:
  - POST to: `{HIKCENTRAL_BASE_URL}{HIKCENTRAL_QR_CREATE_PATH || "/qr-codes"}`
  - Optional auth header: `Authorization: Bearer {HIKCENTRAL_API_KEY}`
  - Response must include `qrId` and `qrImageBase64` (or under `data`)

## Configuration

Environment variables:

- `HIKCENTRAL_MOCK`
- `HIKCENTRAL_BASE_URL`
- `HIKCENTRAL_QR_CREATE_PATH` (default: `/qr-codes`)
- `HIKCENTRAL_API_KEY` (placeholder auth mechanism)
- `QR_ENFORCE_SINGLE_ACTIVE` (default: true; set to `false` to allow multiple active QRs per type/unit)
